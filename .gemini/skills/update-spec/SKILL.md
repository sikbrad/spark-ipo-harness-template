# 명세·업무규칙 업데이트

프로그램 명세 또는 업무규칙이 변경될 때 `proc/spec/`의 관련 문서를 업데이트한다.

## 디렉토리 구조

`proc/spec/` 하위는 카테고리별로 분리한다. 없으면 필요할 때 생성한다.

| 폴더 | 용도 |
|------|------|
| `FSD/` | Functional Spec Document — 기능 단위 명세 |
| `rules/` | 업무규칙(비즈니스 로직, 권한, 컨벤션) |
| `architecture/` | 아키텍처/구성요소/의존 관계 |
| `nfr/` | 비기능 요구사항(성능·보안·운영) |
| `user-flow/` | 사용자 흐름·시나리오 |
| `api/` | API 스펙·계약 |
| `work-rules/` | 작업 규칙·체크리스트 |
| `decisions/` | ADR(Architecture Decision Record) |

## 절차

1. `proc/spec/` 내 기존 문서를 확인한다 (위 카테고리 우선 탐색)
2. 변경 사항과 관련된 문서를 식별한다
3. 해당 문서를 수정한다. 관련 문서가 없으면 새로 생성한다
4. 영향받는 다른 문서도 함께 동기화한다 (예: FSD 변경 → work-rules 체크리스트)
5. 변경 내용을 유저에게 요약 보고한다

## 업무규칙 변경 시 추가 절차

업무규칙(비즈니스 로직, 세금, 가격, 상태 전이, 채번, 수금, 편집권한, 권한 등) 변경인 경우:

1. `rules/` 내 해당 도메인 문서를 **우선** 업데이트한다
   - 비즈니스 로직 통합 정리 문서가 있다면 그 문서에 반영
   - 없으면 도메인별로 `rules/<domain>_rules.md` 생성
2. 권한·편집권한 변경이면 `rules/` 내 권한 설계/편집권한 문서도 동기화한다
3. 관련 FSD 문서와 `work-rules/` 체크리스트도 동기화한다
4. **규칙 문서를 먼저 업데이트한 후** 코드를 수정한다

## ADR 처리

명세 변경이 아키텍처 결정 수준(기술 선택, 통합 방식, 트레이드오프)이면 `decisions/`에 `ADR-NNN_제목.md` 형식으로 생성한다.

ADR 항목: 컨텍스트 / 결정 / 대안 / 결과 / 상태(Proposed·Accepted·Superseded).

## 규칙

- 기존 문서가 있으면 수정, 없으면 생성한다
- 코드를 수정하지 않는다 — 문서만 다룬다
- 한글로 작성한다
- 변경 시 영향받는 다른 문서도 함께 업데이트한다
- `proc/archive/`는 열람·수정하지 않는다 (CLAUDE.md 규칙)
- `$ARGUMENTS`에 변경 내용이 들어온다
