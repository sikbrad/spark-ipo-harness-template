# 사전 조사/연구 문서 생성·업데이트

구현 전 주제에 대해 현황을 분석하고, 방안을 비교하고, 구현 제안을 포함한 리서치 문서를 작성·업데이트한다.

## 출력 위치: `proc/research/`

## 파일명: `YYYYMMDD_주제요약.md`

## 워크플로

1. **현황 파악**: 관련 코드베이스 탐색 (컨트랙트, 서버, 프론트엔드, 스키마)
2. **문제 정의**: 현재 동작과 문제점을 구체적 시나리오로 정리
3. **방안 비교**: 최소 2개 이상의 해결 방안을 장단점과 함께 비교
4. **구현 제안**: 권장안의 레이어별 구현 방향 (DB → API → Server → Frontend)
5. **문서 작성**: 아래 구조에 따라 `proc/research/`에 저장

## 문서 구조

```markdown
# {조사 제목}

> **도메인**: {도메인 코드} (예: ORD, COM)
> **작성일**: YYYY-MM-DD
> **상태**: 리서치/제안

---

## 1. 현재 문제
문제 상황 설명 + 구체적 시나리오

### 원인 분석
레이어별 현 상태 테이블

## 2. 방안 비교
### Option A: {방안명} ⭐ 권장 (권장안에만 표시)
장단점 테이블

### Option B: {방안명}
장단점 테이블

## 3. 권장안: {선택한 방안}
### 구현 범위
레이어별 변경 사항 + 코드 예시

## 4. 적용 대상
영향 범위 테이블

## 5. 구현 단계
Phase별 점진 적용 계획

## 6. 리스크 및 고려사항
마이그레이션, 호환성, 엣지케이스 등
```

## 규칙

- 결과물은 반드시 `proc/research/`에 저장한다 (SPARK의 R)
- `proc/plan/`에는 저장하지 않는다 (계획 문서 전용)
- `proc/spec/`에는 저장하지 않는다 (설계 명세 전용)
- 기존 문서가 있으면 새 파일을 만들지 말고 해당 문서를 업데이트한다
- 코드베이스를 충분히 탐색한 뒤 현황을 파악한다 (추측 금지)
- 방안은 최소 2개 이상 비교하되, 권장안을 명확히 표시한다
- 구현 제안에는 레이어별(DB/API/Server/Frontend) 코드 예시를 포함한다
- 도메인 코드 체계(ORD, COM 등)를 활용하여 도메인을 명시한다
- `$ARGUMENTS`에 조사 주제가 들어온다
