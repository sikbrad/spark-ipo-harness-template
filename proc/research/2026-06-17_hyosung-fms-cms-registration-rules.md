# 효성 FMS CMS 거래 등록 업무 규칙 리서치

작성일: 2026-06-17

## 결론

효성 CMS+ 등록은 단순히 결제수단만 입력하는 작업이 아니다. DOF 포털의 주문/계약 정보를 먼저 확인하고, 그 식별자를 효성 CMS+의 `회원번호`와 `계약정보`에 맞춰 넣은 뒤, 결제수단이 CMS 계좌인지 카드인지에 따라 인증/동의/세금계산서 처리가 갈린다.

핵심 규칙은 아래다.

- `회원번호`는 DOF 포털의 주문번호 또는 계약번호와 맞춘다. 대소문자를 구분한다.
- 장비 일시 판매는 주문번호 계열(`OD...`)로도 관리 가능하지만, 렌탈은 계약번호 계열(`CO...`/`C...`)로 관리한다.
- 렌탈 회원은 효성 CMS+의 회원/계약 구분을 `렌탈`로 분류한다.
- CMS 계좌는 은행, 예금주명, 계좌번호, 예금주의 생년월일 또는 사업자등록번호, 동의자료가 필요하다.
- 법인 계좌/법인카드는 사업자등록번호를 넣고, 개인 계좌/개인카드는 생년월일을 넣는다.
- 카드는 카드번호, 유효기간, 명의자, 생년월일/사업자등록번호를 넣고 `카드인증`을 한다. 별도 동의서 업로드 없이 즉시 연결된다.
- 카드 결제 건은 세금계산서를 발행하지 않는 흐름으로 설명됐다. 계좌 기반 신규 렌탈이면 세금계산서 발행 흐름으로 설명됐다.
- 기존 출금 계좌 변경은 신규 회원등록이 아니다. 기존 계약정보를 먼저 `중지`하고, 새 계좌 계약정보를 `추가`한다.

## 업무 흐름

### 1. DOF 포털에서 기준 주문/계약을 찾는다

영상에서는 먼저 DOF 포털에서 `영업관리 > 주문` 또는 `영업관리 > 계약`으로 이동해 대상 주문/계약을 찾는다.

확인해야 하는 값:

| 포털 값 | 효성 CMS+ 사용처 |
|---|---|
| 주문번호 | 장비 판매 건의 `회원번호` 후보 |
| 계약번호 | 렌탈 건의 `회원번호`/계약 연결 기준 |
| 고객사/고객명 | `회원명`, 예금주/명의자 검증 |
| 사업자등록번호 | 법인 계좌/법인카드의 `생년월일/사업자등록번호` 칸 |
| 휴대전화 | `납부자 휴대전화`, 간편동의/청구서 발송 수단 |
| 상품명/상품구분 | 효성 CMS+ `상품 정보` |
| 금액 | `상품금액`, 청구 금액 |
| 납부일/약정일 | 효성 CMS+ `약정일` |
| 계약기간/렌탈기간 | 효성 CMS+ `청구기간` |

### 2. 효성 CMS+ 회원번호를 정한다

효성 CMS+의 `회원 등록` 화면에서 `회원번호`는 DOF 포털 식별자와 맞춘다.

규칙:
- 중복이면 사용할 수 없으므로 회원번호 중복 조회가 필요하다.
- 대소문자를 구분한다.
- 장비 판매 건은 주문번호(`OD...`)를 써도 된다고 설명됐다.
- 렌탈은 계약번호(`CO...` 또는 `C...`)로 관리한다.
- 렌탈은 나중에 구분/회계 처리를 위해 회원/계약 구분을 `렌탈`로 설정한다.

영상 근거:
- 카드 신규등록 전사 00:50-01:36: CMS 회원 코드, `OD`는 주문번호, `CO`는 계약번호, 렌탈은 `C`로 관리한다는 설명.
- 계좌변경 전사 00:30-00:52, 02:51-02:59: 회원번호는 Sales/Sage 계열 번호와 매칭되고 대소문자를 구분한다는 설명.

### 3. 계약/상품 정보를 입력한다

효성 CMS+의 계약정보/상품정보에는 포털 주문 또는 계약의 상품·금액·기간을 옮긴다.

입력 규칙:
- 상품명은 포털의 상품 성격과 맞춘다. 예: 장비는 `CRAFT`, 워런티는 워런티 상품, 렌탈은 렌탈 상품.
- 회계 구분에 쓰이는 값이므로 상품 구분을 임의로 바꾸지 않는다.
- 기존 계좌 변경이면 기존 계약정보를 복사해 새 계약정보를 만들고, 계좌정보만 바꾸는 방식이 안전하다.
- 금액은 실제 청구 금액과 맞춘다.
- 렌탈 기본기간은 48개월로 설명됐다.
- `약정일`은 납부일과 같은 의미로 설명됐다.

주의:
- 날짜 규칙은 영상상 완전히 표준화되어 있지 않다. 계좌변경 예시에서는 변경 시작일과 마지막 청구 종료일을 맞추고, 약정일 기준으로 청구가 걸린다는 설명이 있었다. 실제 자동화 전에는 `청구기간 종료일` 산정 규칙을 회계 기준으로 별도 확정해야 한다.

### 4. 결제수단이 CMS 계좌일 때

효성 CMS+ `회원 등록` 화면의 `자동결제 공통정보 > 결제수단`에서 `CMS`를 선택한다.

화면 필드:
- `은행`
- `예금주명`
- `계좌번호`
- `생년월일/사업자등록번호`
- `동의자료` 파일 업로드
- `납부자 휴대전화`
- `납부자 이메일`
- `청구서 발송 수단`

입력 규칙:
- 법인통장이면 `생년월일/사업자등록번호` 칸에 사업자등록번호를 넣는다.
- 개인계좌면 예금주의 생년월일을 넣는다.
- CMS 계좌는 동의자료가 필요하다. 계좌변경 영상에서는 `디오에프연구소 CMS 자동이체신청서` PDF를 열어 확인했다.
- 계좌 연결은 즉시가 아니라 약 이틀 걸린다는 설명이 있었다.

계좌변경일 때:
- 기존 회원을 새로 등록하지 않는다.
- 기존 회원의 `계약정보 수정`으로 이전 계좌 계약을 중지한다.
- 그래야 기존 계좌에서 더 이상 출금되지 않는다.
- 이후 `계약정보 추가`로 새 계좌 정보를 추가한다.
- 기존 계약정보를 복사해 상품/금액/기간을 유지하고 계좌정보만 바꾸는 흐름이 설명됐다.

### 5. 결제수단이 카드일 때

효성 CMS+ `자동결제 공통정보 > 결제수단`에서 `카드`를 선택한다.

화면 필드:
- `카드번호`
- `유효기간`
- `명의자`
- `생년월일/사업자등록번호`
- `카드인증`
- `동의방식`
- `동의 요청수단`
- `납부자 휴대전화`
- `청구서 발송 수단`
- `청구 자동생성`
- `청구서 자동발송`
- 증빙: `미발급`, `현금영수증`, `세금계산서`

입력 규칙:
- 법인카드는 사업자등록번호를 넣는다.
- 개인카드는 생년월일을 넣는다.
- 납부자 휴대전화는 동일하게 넣는다.
- 카드 건은 계산서 발행이 필요 없는 흐름으로 설명됐다.
- 카드 결제는 동의서 파일 업로드를 하지 않는다.
- `카드인증`을 누르고 오류가 나면 바로 안 되는 카드로 본다.
- 카드 인증이 통과되면 회원 등록 시점부터 바로 연결된다.

영상 근거:
- 카드 신규등록 전사 03:41-03:58: 신용카드는 세금계산서를 발행하지 않는다고 설명.
- 카드 신규등록 전사 05:42-06:22: 법인카드/개인카드 식별번호, 휴대전화, 동의서 업로드 불필요, 카드인증, 즉시 연결 규칙.

## 자동화로 옮길 때 필요한 판단 규칙

### 회원번호 선택

자동화는 포털 레코드 유형을 먼저 판단해야 한다.

- 계약/렌탈이면 계약번호를 우선 사용한다.
- 일반 장비 주문이면 주문번호를 사용할 수 있다.
- 회원번호 생성 전에는 효성 CMS+에서 중복 조회가 필요하다.
- 포털 식별자 원문 대소문자를 그대로 보존한다.

### 결제수단 분기

결제수단별 필수값이 다르다.

- `CMS`: 은행, 계좌번호, 예금주, 생년월일/사업자등록번호, 동의자료 파일.
- `카드`: 카드번호, 유효기간, 명의자, 생년월일/사업자등록번호, 카드인증.

카드와 계좌의 승인 상태도 다르다.

- 카드: 인증 성공 시 즉시 연결.
- CMS 계좌: 연결까지 약 이틀 걸림.

### 개인/법인 분기

예금주/카드 명의가 개인이면 생년월일, 법인이면 사업자등록번호를 넣는다.

이 규칙은 계좌와 카드 모두 동일하게 설명됐다.

### 세금계산서 분기

영상 기준 분기:

- 카드 결제: 세금계산서 발행하지 않음.
- 계좌 기반 신규 렌탈: 세금계산서 발행 흐름.
- 기존 계좌변경 예시는 `세금계산서 미발급` 케이스로 표시됐다.

자동화 전에는 카드/계좌/렌탈/장비판매/기존변경 각각의 증빙 정책을 회계 기준으로 확정해야 한다.

## 남은 확인사항

1. `Sage Number`라고 전사된 값이 현재 포털에서 정확히 어떤 필드명인지 확인해야 한다. 영상 맥락상 주문번호/계약번호/세일즈 번호 계열 식별자다.
2. `청구기간 종료일`을 어떤 기준으로 산정할지 회계 기준이 필요하다. 영상에는 48개월 렌탈과 기존 계좌변경 종료일 예시가 나오지만 범용 공식은 분리되어 있지 않다.
3. `청구서 자동발송`은 등록 화면의 토글과 상세 화면의 `수동` 표시가 같이 관찰된다. 자동화 전 기본값을 확정해야 한다.
4. 증빙 정책은 자동화에 넣기 전에 회계가 승인한 매트릭스로 고정해야 한다.
5. 계좌 연결까지 약 이틀 걸리는 상태를 포털에 어떻게 표시할지 결정해야 한다. 예: `신청중`, `승인대기`, `연결완료`, `오류`.

## 추천 자동화 구조

1. 포털에서 주문/계약 레코드를 읽고 CMS 등록 후보 payload를 만든다.
2. 레코드 유형으로 회원번호 규칙을 적용한다.
3. 결제수단이 카드인지 CMS 계좌인지 분기한다.
4. 개인/법인 식별번호 타입을 결정한다.
5. 세금계산서/증빙 정책을 결정한다.
6. 효성 CMS+ 등록 전 dry-run 화면 또는 검토 payload를 만든다.
7. 등록 후 효성 회원번호, 계약번호, 결제수단 상태, 인증/신청 결과를 포털에 다시 기록한다.

자동화에서 가장 위험한 부분은 등록 클릭 자체보다 식별자/증빙/기간/기존계좌 중지 여부다. 특히 계좌변경은 기존 계약을 먼저 중지하지 않으면 이전 계좌에서 계속 출금될 수 있으므로 별도 안전장치가 필요하다.
