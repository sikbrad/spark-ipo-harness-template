## 정리 원칙

1. 최상위 폴더 구조는 유지한다.
2. `public`, `협업`, `보안`, `기밀` 경계를 넘기는 이동은 기본적으로 금지한다.
3. `Inbox(협업)`은 임시 수집함으로만 둔다. 완료/장기 보관 문서는 `프로젝트`, `리서치`, `지식`, `회의록` 중 하나로 이동한다.
4. 날짜별 회의록은 중복이 아니라 시리즈로 본다. `AX팀회의`, `영업정기회의+AX`, `정기회의`는 병합 대상이 아니다.
5. `휴지통(보안)`은 삭제 전 보관소로 보고, 실제 삭제나 복구는 별도 검수 후 결정한다.
6. Confluence 실행 시에는 본문 재생성 방식이 아니라 move/copy native 기능을 써야 한다. 기존 아이콘, 첨부, 관련 콘텐츠, 권한, 링크 손상을 피하기 위해서다.

### P2. 프로젝트(협업) 하위 구조 보강

`프로젝트(협업)`은 `ING 프로젝트 (협업) / [AX-129] 주문앱` 아래에 실제 본류가 모여 있다. 구조 자체는 유지할 만하지만, 일부 하위가 깊고 로그/기획/회의/인터뷰가 섞여 있다.

- `[AX-129] 주문앱 / 연동시스템`

- `[AX-129] 주문앱 / 개발로그`

이미 존재하는 `주문웹 기획`, `[AX-137] 주문 및 물류 사용자 인터뷰`, `주문웹 연동시스템`, `주문챗봇`은 유지하고, Inbox에서 들어오는 개발로그/현장테스트만 자연스럽게 붙이는 방식이 좋다.

### P3. 지식(public) 큐레이션

`지식(public)`에는 `바이브코딩 노하우 쌓기 / 하위 AI 생성문서`가 있고, 이 하위에 AI 생성문서 20개가 있다. 상위 문서는 지식 축적 의도에 맞지만, 생성 원문 전체가 곧바로 public 지식으로 노출되는 구조는 품질 관리가 어렵다.

제안:

- `바이브코딩 노하우 쌓기 / 초안-AI생성원문` 식으로 초안 성격을 명시한다.
- 실제로 재사용 가능한 문서만 별도 curated 페이지로 승격한다.
- public 공개 의도와 내부 회사 AX 사례가 섞여 있으면 `지식(보안)`으로 분리한다.

### P4. 휴지통(보안) 정리

`휴지통(보안)`은 141개 노드로 가장 크다. 실제 삭제보다는 `legacy 프로젝트구조 (보안)`와 테스트/템플릿/Task류가 한데 있는 상태다.

권장 운영:

- `기본템플릿원래있던거`, `한번에지우자`, `legacy 프로젝트구조 (보안)`은 당장 삭제하지 않는다.
- `Template - ...`, `sample`, `[Task] TestTest*`는 삭제 후보로 분리 검토한다.
- `legacy 프로젝트구조 (보안)` 안의 AX 프로젝트 문서는 과거 이력으로 가치가 있으므로, 삭제보다 `보관/legacy` 상태를 명확히 하는 편이 안전하다.
- `AX-54`, `AX-61`처럼 Task와 실제 문서가 중복으로 존재하는 항목은 병합/삭제가 아니라 “Task 문서는 삭제 후보, 실제 문서는 legacy 보관”으로 나누는 게 좋다.

## 실제 실행 전 체크리스트

실행은 아직 하지 않았다. 실행 단계로 넘어가려면 아래가 먼저 필요하다.

1. 현재 스냅샷을 다시 떠서 변경 전 기준을 고정한다.
2. `source_id`, `source_path`, `target_path`, `action`, `risk`, `reason` 컬럼을 가진 move plan CSV/JSON을 만든다.
3. `public`/`협업`/`보안`/`기밀` 경계를 넘는 항목은 별도 승인 대상으로 표시한다.
4. 첨부, 권한, 레이블, child count가 유지되는지 dry-run 또는 소량 샘플로 검증한다.
5. 이동은 P0/P1의 작은 묶음부터 시작한다.
6. 각 batch 후 API readback으로 parent, child count, URL 접근 가능성을 확인한다.
7. 최종 보고에는 이동 전/후 경로와 실패/보류 항목을 남긴다.
