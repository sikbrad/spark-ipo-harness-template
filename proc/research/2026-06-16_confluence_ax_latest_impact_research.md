# AX 최신 문서 기반 임팩트 리서치

작성일: 2026-06-16
작성 목적: 최근 2주 이내 AX 스페이스 문서와 지정 문서를 읽고, 지금 사용자가 무엇을 하면 가장 임팩트가 클지 판단한다.

## 결론

가장 임팩트가 큰 일은 **수금/미수/거래원장/고객 발송/입금확인 루프를 하나의 운영 제품으로 닫는 것**이다.

지금 AX 문서들은 여러 주제를 다루지만, 반복해서 같은 문제로 수렴한다. 영업은 주문을 쉽게 넣고 싶어 하고, 회계/경영은 미수와 연체를 줄이고 싶어 하며, 고객에게 보내는 거래내역서/운송정보/원장 발송은 이미 실제 돈 회수와 연결되어 있다. 주문웹/챗봇, 가격서버, 재고, 계좌조회, 카카오/이메일 발송은 따로 떨어진 기능이 아니라 이 루프를 완성하는 부품으로 봐야 한다.

따라서 사용자가 지금 해야 할 역할은 개발 목록을 더 늘리는 것이 아니라, **"돈이 들어오고 업무가 줄어드는 하나의 end-to-end 흐름"을 정의하고, 팀이 그 흐름만 먼저 끝내게 만드는 것**이다.

## 패턴 분석

### 1. 요구는 많지만 한 가지 운영 루프로 묶인다

최근 문서에 등장한 기능명은 많다. 주문웹, 챗봇, 가격서버, 고객등급, 재고, 계좌조회, 거래내역서, 운송정보, 팀즈 알림, AX01 자동화, 기술지원 이관까지 흩어져 있다.

하지만 실제 업무 흐름으로 보면 중심은 다음 하나다.

1. 고객/주문/계약/등급/미수 상태를 정확히 본다.
2. 주문이 들어올 때 위험 조건을 알려 준다.
3. 거래원장/거래명세서/운송정보를 고객에게 보낸다.
4. 고객이 봤는지, 입금했는지, 다음 조치가 필요한지 기록한다.
5. 회계/영업/주문 담당자가 다음 행동을 놓치지 않게 알린다.

이 루프가 닫히면 매출 회수, 영업 부담, 주문 운영, 고객 커뮤니케이션이 동시에 좋아진다. 반대로 이 루프 없이 각 기능을 따로 만들면 "기능은 늘었는데 확인할 일이 더 늘어나는" 상태가 된다.

### 2. 사용자 불편은 입력보다 확인과 후속조치에서 생긴다

`260615 생각정리`에는 사람이 한 번 작성하고 떠난 뒤, 시스템이 그 사람에게 계속 확인을 요구하는 문제가 적혀 있다. 이 말이 중요하다. AX가 해야 할 일은 입력 폼을 더 똑똑하게 만드는 것에 그치지 않고, **작성 이후의 확인/알림/후속 처리 부담을 줄이는 것**이다.

그래서 주문웹/챗봇 UX 개선도 단순 검색 품질 문제가 아니다. 제품 후보가 틀리면 주문 담당자가 확인해야 하고, 거래처가 애매하면 영업이 다시 확인해야 하며, 주소가 틀리면 물류가 멈춘다. 결국 후속확인 비용을 줄이는 방향으로 설계해야 한다.

### 3. 수금/미수는 경영 임팩트가 가장 직접적이다

최근 문서에는 미수, 연체, 고객등급, 수금계획, 거래원장 발송, 계좌잔액/입금확인, PKG 계약잔액 보정이 반복된다. 특히 거래원장을 보낸 뒤 돈이 들어왔다는 회의 메모는 중요한 신호다. 이 영역은 "편해진다"와 "돈이 빨리 들어온다"가 동시에 성립한다.

여기서 사용자가 가장 크게 기여할 수 있는 부분은 코드를 직접 더 많이 쓰는 것이 아니라, 회계/영업/개발 사이의 기준을 확정하는 것이다.

- 어떤 고객을 고위험/주의로 볼 것인가
- 주문 진행을 막는 조건과 경고만 주는 조건은 무엇인가
- 수금계획은 고객 귀속인가, 주문 귀속인가
- 거래원장에 최근 입금내역을 어디까지 넣을 것인가
- 고객에게 보낸 링크의 유효기간, 비밀번호, 법적 검토 기준은 무엇인가
- 발송 성공/열람/입금/후속 연락을 어떤 로그로 남길 것인가

이 기준이 없으면 개발팀은 계속 개별 기능 요청을 처리하게 된다.

### 4. 주문웹/챗봇은 adoption 병목이다

`6~7월 해야할 것`과 `260611 영업 피드백 회의`에서 제품 후보, 제품정보 구조화, 가격서버, 제품 제외 기능, 색상 구분, 검색/리젝 조건이 반복된다. 이는 주문앱 사용 빈도 저하와 직접 연결된다.

다만 이 영역은 수금 루프보다 넓고 깊다. 제품 마스터, 가격, 고객사/주소/사업자 검증, 재고까지 이어지기 때문이다. 그래서 모든 것을 한 번에 잡기보다, 수금/발송 루프에 필요한 주문 품질 병목부터 좁혀야 한다.

우선순위가 높은 주문웹 개선은 아래 정도다.

- 매칭 안 된 제품을 제외하고 주문을 올릴 수 있게 하는 흐름
- 후보 선택 시 사람이 틀리지 않도록 제품 속성/사이즈/직경을 더 선명하게 보여 주는 UI
- 가격 조회와 총액 계산
- 고객사/사업지주소 검증
- 주문에 수금계획 섹션 연결

### 5. 기술지원/상담은 중요하지만 지금 1순위는 아니다

기술지원 문서들은 자료검색, 응대이력, 상담 도구, 티켓 관리, 채널톡 전환 같은 큰 주제를 담고 있다. 임팩트는 크지만, 기존 업무 습관과 도구를 바꾸는 성격이라 변경관리 리스크가 높다.

지금 이걸 1순위로 잡으면 "자료를 어디로 옮길 것인가", "상담툴을 무엇으로 바꿀 것인가" 같은 의사결정에 시간이 많이 쓰인다. 지금은 기술지원 전체 재구축보다, 포탈/계약/렌탈/제품사진처럼 이미 주문·고객 데이터와 연결되는 항목만 수금/고객 커뮤니케이션 루프에 붙이는 편이 낫다.

## 추천 우선순위

### P0. 수금/거래원장/고객발송 루프를 2주짜리 대표 AX 성과로 만든다

이것이 가장 임팩트가 크다.

목표는 "거래원장 PDF를 만들 수 있다"가 아니라, **미수/연체 고객에게 필요한 자료를 보내고, 열람/입금/후속조치까지 추적하는 운영 루프**다.

추천 범위:

- Top 20 미수/연체 고객 리스트 확정
- 고객등급/PQRS 또는 주의/고위험 기준 확정
- 거래원장/연체·미수 내역 PDF 생성
- 이메일 첨부 + 카카오/문자 URL 발송
- 발송 로그와 열람 여부 기록
- 주문 단위 수금계획 입력
- 입금확인 1차는 엑셀 업로드 또는 수동 보정 UI로 시작
- 팀즈 알림은 쿨타임을 두고 "사람이 다음 행동을 해야 할 때"만 보냄

성공 기준:

- 거래원장 발송 대상 고객 수
- 발송 후 열람률
- 발송 후 입금 또는 회신된 금액
- 회계/영업이 수동으로 보내던 메시지 수 감소
- 주문 진행 중 "미수 확인 때문에 되돌아가는" 건수 감소

### P1. 주문웹/챗봇 UX는 "주문 담당자의 확인 부담 감소" 기준으로 자른다

제품검색과 챗봇 개선은 중요하지만, 목적을 "AI 주문 정확도"로 두면 너무 넓어진다. 목적을 "주문 담당자가 다시 확인해야 하는 건수 감소"로 잡아야 한다.

우선 작업:

- 제품 후보 표시에 속성/사이즈/직경/카테고리 강조
- 제품 임베딩 단계에서 구조화 속성 생성
- LLM 필터링에 검색 조건뿐 아니라 리젝 조건 추가
- 매칭 안 된 제품 제외 후 주문 생성
- 가격 조회/총액 계산/배송비 기본 항목

성공 기준:

- 챗봇 주문 중 수동 재확인 건수
- 매칭 실패 품목 수
- 주문 생성까지 걸리는 시간
- 주문 담당자가 되돌려 보낸 주문 수

### P2. 계좌잔액/입금확인은 직접 스크래핑보다 "신뢰 가능한 입력 경로"를 먼저 만든다

계좌잔액 자동조회 문서에서는 직접 자동화가 인증/보안 단계에서 막힌 것으로 보인다. 이 영역은 무리한 브라우저 자동화보다 외부 API/기존 서비스/엑셀 업로드를 비교해서 안전한 경로를 정해야 한다.

추천:

- 1차: 수금 엑셀 업로드와 수동 입력 UI로 운영 루프를 먼저 닫는다.
- 2차: 이미 회사에서 쓰는 서비스 또는 은행/계좌 API 후보를 비용/권한/지원범위로 비교한다.
- 3차: SMS/앱/브라우저 자동화는 보안·운영 책임자가 명확할 때만 붙인다.

이유는 간단하다. 입금 자동화가 늦어져도 거래원장 발송과 후속조치 루프는 먼저 성과를 낼 수 있다.

### P3. 기술지원은 "전체 지식이관"이 아니라 "상담 로그와 고객/제품 데이터 연결"부터 한다

기술지원은 자료가 파편화되어 있고 상담 도구 전환까지 걸려 있다. 지금 당장 할 일은 전체 RAG나 전체 이관이 아니다.

작게 시작할 수 있는 범위:

- 고객/장비/렌탈 계약/제품사진을 포탈 고객 화면에서 볼 수 있게 연결
- 상담 중 필요한 링크 모음만 정리
- 상담/응대 이력을 고객사와 연결하는 최소 기록 필드 정의
- 채널톡 등 도구 전환은 별도 의사결정 문서로 분리

## 사용자가 지금 해야 할 일

### 1. 이번 주 대표 과제를 하나로 선언한다

추천 문장:

> 이번 주 AX 대표 과제는 "미수/거래원장/고객 발송 루프를 닫아서 실제 입금과 후속조치가 보이게 만드는 것"으로 잡는다.

이렇게 선언해야 개발 큐가 흩어지지 않는다.

### 2. 회계/영업과 기준 6개를 확정한다

회의에서 확정해야 할 질문:

- 고위험/주의 고객 기준은 무엇인가
- 주문을 막는 조건과 경고만 주는 조건은 무엇인가
- 수금계획은 고객 단위인가, 주문 단위인가
- 거래원장에 최근 입금내역을 몇 개월 포함할 것인가
- 고객에게 보내는 링크는 유효기간/비밀번호/법적 문구를 어떻게 할 것인가
- 발송 후 열람/회신/입금 상태를 누가 보고 후속조치할 것인가

이 6개가 확정되면 개발 항목은 자연스럽게 줄어든다.

### 3. 개발팀에는 "기능 목록"이 아니라 "하루짜리 운영 시나리오"를 준다

예시 시나리오:

1. 회계가 Top 20 미수 고객을 본다.
2. 한 고객을 선택해 거래원장/미수내역을 생성한다.
3. 영업 또는 회계가 이메일/카카오 URL로 발송한다.
4. 시스템은 발송 로그와 열람 여부를 남긴다.
5. 입금 또는 수금계획이 등록된다.
6. 주문 진행 시 해당 상태가 경고 또는 차단 조건으로 반영된다.
7. 미처리 상태는 팀즈로 한 번만 알린다.

이 시나리오가 되면 주문웹, 거래원장, 발송, 수금, 알림이 한 흐름으로 붙는다.

### 4. AX01/팀즈봇/아마란스 자동화는 "실행 엔진"으로만 본다

AX01, 팀즈봇, Amaranth 자동화는 매우 유용하지만 지금은 주인공이 아니다. 주인공은 업무 루프다. 실행 엔진은 사람이 정한 기준과 로그 구조가 있을 때만 안정적으로 붙는다.

운영 원칙:

- 자동 실행 전에는 항상 원본/근거/로그가 있어야 한다.
- 되돌림이 필요한 작업은 수동 승인 또는 dry-run을 둔다.
- 브라우저 밖 알림은 쿨타임을 둔다.
- 자동화가 실패해도 사람이 이어받을 fallback 화면이 있어야 한다.

## 하지 않는 편이 좋은 일

- 카톡방 같은 공통 공간을 먼저 직접 구현하지 않는다. 문제 정의는 맞지만 범위가 너무 커진다.
- 기술지원 전체 자료를 한 번에 RAG/새 도구로 옮기지 않는다. 변경관리와 신뢰 문제가 크다.
- 은행/계좌 브라우저 스크래핑에 먼저 매달리지 않는다. 인증/보안 벽이 이미 확인됐다.
- "바로발송"을 독립 편의기능으로 먼저 만들지 않는다. 거래원장/수금 루프와 붙일 때 가치가 커진다.
- 모든 주문웹 UX를 한꺼번에 고치려 하지 않는다. 주문 담당자의 재확인 건수를 줄이는 항목부터 한다.

## 2주 실행안

### 1주차: 기준 확정과 얇은 루프

- Day 1: 회계/영업 기준 6개 확정
- Day 2: Top 20 미수/연체 고객과 테스트 고객 3곳 선정
- Day 3: 거래원장/미수내역 PDF와 고객 발송 템플릿 확정
- Day 4: 발송 로그/열람 로그/수금계획 최소 스키마 확정
- Day 5: 이메일 또는 카카오 URL 중 하나로 먼저 end-to-end 테스트

### 2주차: 주문/알림/입금 연결

- Day 6: 주문 화면에 수금/주의 상태 표시
- Day 7: 수금계획 입력과 상태 변경 연결
- Day 8: 엑셀 업로드 또는 수동 입력으로 입금확인 반영
- Day 9: 팀즈 알림을 쿨타임 포함해 연결
- Day 10: 회계/영업과 실제 발송 1회, 회고, 다음 배치 정의

## 최종 제안

사용자는 지금 **AX 제품 오너처럼 행동하는 것**이 가장 좋다. 코드를 더 많이 쓰기보다, 회계/영업/개발이 같이 보는 하나의 업무 루프를 잡고, 기준을 확정하고, 2주 안에 실제 돈/시간 절감 지표가 보이게 해야 한다.

한 문장으로 줄이면:

> "거래원장을 만들고 보내는 기능"이 아니라 "미수 고객에게 필요한 자료를 보내고, 열람/입금/후속조치까지 남는 수금 운영 루프"를 만든다.

이 방향이 최근 AX 문서들이 가리키는 가장 큰 교집합이고, 회사 입장에서 가장 설명하기 쉬운 성과다.
