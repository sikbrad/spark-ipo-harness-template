# 포탈 주문취소 데이터 변경 규칙 수립 계획

## 목표

포탈 주문 상태 `CLOSED_LOST`의 사용자 표시명 `주문취소` 전환에 맞춰, 주문취소 이후 데이터 변경 가능 범위와 수정/승인 권한자를 정의한다.

완료 기준:
- 주문취소를 단순 상태값 수정이 아니라 사유, 승인, 감사로그가 남는 업무 이벤트로 정의한다.
- 상태별 취소 가능 조건, 승인자, 취소 후 수정 가능 필드를 확정한다.
- 포탈 명세와 구현 대상이 분리된 체크리스트로 정리된다.

## 현재 확인된 사실

- 포탈 주문 상태에는 `CLOSED_LOST`가 있고 화면 라벨은 현재 `실주`, 별도 계획에서는 `주문취소`로 표시명 변경을 준비 중이다.
- `CLOSED_LOST`는 서버/프론트 상태 전이 그래프에서 종결 상태로 되어 있고, 현재 상태에서 나가는 전이가 없다.
- 서버의 `canEditOrder()`는 `CLOSED_LOST`를 명시 허용하지 않아 기본적으로 수정 불가다.
- 프론트 필드 잠금 규칙은 `CLOSED_LOST`를 `COMPLETED`와 유사한 종결 상태로 주석 처리했지만, 서버와 완전히 같은 방식으로 닫는지는 추가 확인이 필요하다.
- 과거 인터뷰 자료에는 주문취소 규칙 부재, 사유 필수, 유예 기간, 당일 취소 제한, 대형 주문의 선입금/위약금 필요성이 이미 문제로 남아 있다.

## 권장 방향

주문취소 권한은 `write` 일반 수정권한에 묶지 않는다. 별도 capability를 두고, 취소 요청자와 승인자를 분리한다.

초안:
- `cancel.order.request`: 담당 영업/영업팀장이 취소 요청 생성. 사유, 고객 요청 근거, 취소 영향 입력 필수.
- `cancel.order.approve`: 영업팀장 또는 운영관리자가 승인. 요청자 본인은 승인 불가.
- `cancel.order.logistics`: 출고승인 이후 주문취소 승인에 물류 확인 필수.
- `cancel.order.accounting`: 수금, 계산서, ERP, 회계마감 영향이 있는 주문취소 승인에 회계 확인 필수.
- `cancel.order.force`: 예외 복구/오입력 정정용. ADMIN + `approve.done` 또는 지정 운영관리자에게만 부여.

화면 권한 매트릭스:

| 권한/역할 | 가능 | 불가능 |
|---|---|---|
| 담당 영업 | 주문취소 요청, 사유 입력, 고객 요청일 입력, 고객 요청 증빙 첨부, 요청 철회 | 본인 요청 승인, 취소 확정, 제품/수량/단가/고객사/수금/출고 정보 수정 |
| 영업팀장/운영관리 | 취소 승인, 반려, 보완요청, 사유 코드 확정, 영향 메모 입력 | 본인이 요청한 건 승인, 물류/회계 영향 있는 건 단독 확정 |
| 물류 | 출고승인 이후 물류 영향 확인, 발주/외주/출하/운송예약 손실 메모 입력, 물류 확인 완료 처리 | 취소 최종 승인 단독 처리, 영업 사유 변경, 수금/계산서/ERP 정보 수정 |
| 회계/수금 | 수금/계산서/ERP 영향 확인, 매출취소 필요 여부 표시, 회계 확인 메모 입력, 수금행 정리 | 취소 영업 사유 변경, 제품/단가/고객사/출고 정보 수정 |
| 관리자/예외처리자 | 오입력 복구, 예외 재개방, 강제 상태 보정, 권한 부여 | 일반 운영 승인자로 상시 사용 |

상태별 초안:
- `REGISTERED`, `APPROVAL_REQUESTED`: 담당 영업이 요청 가능. 금액/제품/수금 영향이 없으면 영업팀장 승인으로 종결.
- `TEAM_LEAD_APPROVED`, `EXECUTIVE_APPROVED`: 취소 요청은 가능하지만 승인자는 현재 승인 단계보다 상위 또는 동일 승인권자여야 함.
- `RELEASE_APPROVED`, `RELEASE_HOLD`, `SHIPPED`: 물류 확인 필수. 이미 발주/출하/외주 작업이 있으면 취소 수수료/손실 메모 필수.
- `COMPLETED`: 원칙적으로 주문취소 전환 금지. ERP 전송, 자산, 매출/수금이 얽혀 있으므로 기존 `주문완료취소` 절차 또는 별도 회계 반제/반품/매출취소 플로우로 처리.
- `CLOSED_LOST`: 종결 상태. 원 주문 데이터는 수정 불가. 취소 사유/증빙/댓글/해시태그 정도만 append-only로 허용.

## 남은 리스크

- `CLOSED_LOST`가 SF `Closed Lost`에서 온 영업 실패 의미와 실제 주문취소 의미를 같이 쓰고 있어 용어/데이터 의미가 섞일 수 있다.
- 과거에 DB 직접 업데이트로 상태만 바뀐 건은 사유/승인/감사로그가 부족하다. 새 규칙을 소급 적용할지 별도 판단이 필요하다.
- 주문완료 이후 취소는 포탈 단독 상태값 변경으로 끝낼 수 없고 ERP/수금/자산 정합성 검토가 필요하다.
