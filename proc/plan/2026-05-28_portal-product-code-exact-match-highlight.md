# 포탈 목록 첫 번째 컬럼 정확일치 강조 계획

## 목표
목록 검색에서 퍼지/부분 검색 결과는 유지하되, 검색어와 현재 목록의 첫 번째 데이터 컬럼 값이 정확히 일치하는 경우 해당 셀을 눈에 띄게 표시한다.
예: `/orders?view=ord000&q=OD202605201035`에서 `주문번호` 셀 값이 `OD202605201035`이면 그 첫 번째 컬럼 셀을 강조한다.

- 요청자: 이미연
- 사용자 응답 방향:
  - "입력한 코드가 없으면 다른 내역을 안 뜨게" 하는 것은 어렵다.
  - 현재 검색은 확률/유사도 기반 결과를 포함하므로 다른 후보도 떠야 한다.
  - 대신 `정확하게 일치`하는 경우 첫 번째 데이터 컬럼 셀을 빨간색으로 표시하거나 강조한다.

## 가능 여부
가능하다. 검색 결과를 줄이는 서버 로직을 바꾸지 않고, 공통 목록 테이블 렌더링 단계에서 첫 번째 데이터 컬럼 값과 적용된 검색어가 정확히 같은지만 판정하면 된다.

권장 범위는 프론트 중심 변경이다. API 응답 구조, DB 검색 조건, `pg_trgm` 유사도 기준은 유지한다.

- 검색어 상태:
  - `useListPage.ts`가 입력 중 검색어와 실제 적용된 검색어를 분리한다.
  - 화면 강조는 입력 중 값이 아니라 실행된 검색어, 즉 URL `q`/`appliedSearch` 기준이어야 한다.

## 설계
- 정확일치 기준:
  - 기본: `String(firstColumnValue).trim().toLowerCase() === appliedSearch.trim().toLowerCase()`
  - 주문번호/제품코드에서 하이픈/공백 표기 흔들림이 실제로 문제가 되면 보조 비교를 추가한다.
- 표시 방식:
  - 체크박스를 제외한 첫 번째 데이터 컬럼의 셀만 연한 빨강 배경 + 진한 빨강 텍스트/테두리로 강조한다.
  - 셀 안에 작은 `정확일치` 배지를 붙인다.
  - 행 전체 강조보다 셀 강조를 우선한다. 사용자가 비교해야 하는 값은 첫 번째 식별자 컬럼이기 때문이다.
- 검색 결과 정책:
  - 다른 후보는 숨기지 않는다.
  - 정확일치가 없는 경우 별도 차단/필터 없이 빨간 셀이 없는 상태로 둔다.

## 검증 계획
- 정확일치 있음:
  - `/orders?view=ord000&q=<실제 주문번호>`에서 첫 번째 데이터 컬럼인 주문번호 셀만 빨간 강조와 `정확일치` 배지가 표시되는지 확인한다.
- 정확일치 없음:
  - 비슷하지만 완전히 같지는 않은 주문번호로 검색했을 때 유사 결과는 유지되고 빨간 강조는 없는지 확인한다.
- 회귀 확인:
  - 검색어 입력만 하고 검색 실행 전에는 강조가 바뀌지 않아야 한다.
  - 기존 필터/뷰/정렬/다운로드 동작에는 영향이 없어야 한다.

## 리스크
- 공통 `ListPageLayout` 변경이므로 모든 목록 페이지에 적용된다. 단, 강조 조건은 첫 번째 데이터 컬럼의 원본 값과 실행된 검색어의 정확일치로 제한한다.
- 주문번호/제품코드 표기에서 하이픈/공백 차이를 정확일치로 볼지 정책 결정이 필요하다. 1차는 대소문자만 무시하고, 구분자 무시는 후속 정책으로 둔다.
